2. 说到 board2，其实在与MingXi初次沟通时，我们内部以为就是一个普通的数采及看板项目，但随着沟通的深入，我们发现黄埔的专家们有一个更大的愿景，从智能采集到智能展示再到智能分析，最后是智能控制，这是一个完整的闭环，我个人非常欣赏这个愿景，也非常开心能够参与到这个项目中。
3. 数据模型是什么 --> 基于这样一个长远的目标，我们在 Board2 的架构上做了很大的调整。最大的调整就是构建了一个位于边缘的语义数据模型。什么叫语义数据模型呢？我们可以想象在 plc 中的数据，它的数据就是 tag 与 value，而我们这里可以看到，不单单能够表达设备的从属关系，还能把业务专家的关心的业务逻辑也描述进去。比如这里，业务专家可以使用自然语言来描述需要在产线级别统计一个班次总的停机次数，并且描述这个统计信息与哪些数据关联，
4. 使用数据模型做什么 --> 我们在构建 UI 时，就能够自动识别该规则，自动进行计算，不需要黄埔专家浪费宝贵的时间再去构建看板了。当然，现在大家看到的看板还是一个固定的看板，未来，我们是可以基于 AI 去分析语义数据模型，来根据MIngXi 的需求，来动态生成看板，我觉得语义数据模型是实现智能化的基础。
5. 为了实现语义数据模型，我们与黄埔的专家们共创开发了一套系统，这套系统不单单能够构建边缘的语义数据模型，还能够实现更高实时性的数据采集，更高实时性也是我们替换掉 IIH 的原因，因为 IIH 最高只能实现 50ms 的数据采集。
6. 现在语义数据模型有了之后，我们就可以很方便的根据数据模型来搭建看板，这个就是我们 board2 ，这里面的 UI 完全是根据语义数据模型来自动适配的。
7. 我们现在知道语义数据模型可以供给 AI 进行分析，也可以被自动解析生成看板。那么如何创建这个语义数据模型呢？比如，黄埔工厂现在有一条新的产线需要接入系统，未来，可以直接让 AI 来分析 PLC 的结构，再根据业务专家比如泉哥的自然语言描述，来自动生成语义数据模型。但是在未来还没有到了之前，我们也提供了一个方便的工具来帮助大家来构建。这里，只需要填写规则的 Excel，在这里导入，选择 BU，就可以了，可以看到，这个耗时还比较长，因为 FC 这条产线有上千个点位，好，我们看到，语义数据模型已经建立好了。我们点开这里的 Collector control，点击 apply，这个过程实际就是把语义数据模型发送到数据采集器，来自动的更新需要采集的点位。在更新完成之后，我们打开同样的 board 就可以发现，这里根据数据模型自动的添加了一条产线，选择这条产线之后，模型中配置的点位与逻辑也被应用了上来。这里可以看到，这个产线 layout 是默认样式，但是也是能够获取状态的，这里根据需求来编辑 line，保存就可以了。这里，我导入一个上面之前准备好的 layout，这个 layout 是与 line 匹配到，状态会自动绑定。
8. 好，这样，整个流程就通了，通过 Excel 来方便的创建新的产线语义数据模型，自动的更新数据采集器，UI 自动更新。当然，我们也给 mingxi 准备了一些惊喜，她之前总是抱怨管理数据的上下限太麻烦了，特别是根据 sku 来动态决定上下限的时候。在我们的系统中，在 tag 上可以创建 mapping，一些常规的比较，比如这里大于，小于等，都是可以的，也可以添加 and or，这是基于基本规则的。我们在这里还基于语义，做了这样的一个功能，切换到动态 limit，这里选择动态的数据源，选择好之后，在运行时，就会把这个值与下面的 list 进行比较，先匹配到，找到上下限，然后拿当前值与上下限比较，得到是正常的，还是异常的。
